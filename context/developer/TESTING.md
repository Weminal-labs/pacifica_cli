# Testing Strategy

> Generated by setup wizard on 2026-03-29.

## Philosophy

For a 3-day hackathon, testing is pragmatic:
- **Manual testing against Pacifica testnet** is the primary validation method
- **Unit tests** for critical business logic (guardrails, trading math, config parsing)
- **No E2E tests** — time is better spent on features

## Unit Tests (If Time)

Framework: **Vitest** (fast, TypeScript-native, no config needed)

Priority test targets:
1. **Guardrails checker** — spending limits, action whitelist/blacklist, confirmation threshold
2. **Config parser** — valid/invalid YAML, missing fields, defaults
3. **Funding rate math** — APR calculations, spread detection
4. **Number formatting** — price display, PnL formatting, volume abbreviation
5. **Smart order logic** — trailing stop distance calculation, partial TP level math

## Integration Tests

- Test SDK client against Pacifica testnet (real API calls)
- Test MCP tools end-to-end (tool call → SDK → API → response)
- These are manual during hackathon, automated later

## Manual Testing Checklist

Before demo, verify each flow works:
- [ ] `pacifica init --testnet` completes successfully
- [ ] `pacifica scan` shows live data that updates
- [ ] `pacifica trade buy` places an order and shows confirmation
- [ ] `pacifica positions` shows the new position with PnL
- [ ] `pacifica heatmap` renders correctly with active position
- [ ] MCP tools return structured data when called from Claude
- [ ] Agent guardrails block an order that exceeds limits
- [ ] `pacifica agent stop` immediately disables agent trading
- [ ] WebSocket reconnects after brief disconnect

## Coverage Goals

- Hackathon: 0% automated coverage is acceptable if manual testing is thorough
- Post-hackathon: 80%+ for core/ modules, especially guardrails and SDK client
