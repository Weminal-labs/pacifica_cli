# Development Workflow

> Generated by setup wizard on 2026-03-29. Solo developer, 3-day hackathon.

## Branch Strategy

- **`main`** — always deployable (or at least not broken)
- Feature branches optional for larger features: `feat/mcp-server`, `feat/heatmap`
- For a 3-day hackathon, working directly on `main` is acceptable for speed
- If working on a risky feature, branch first

## Commit Style

Conventional commits, short and descriptive:

```
feat: add pacifica scan command with live market data
fix: handle WebSocket reconnection when offline > 5s
refactor: extract shared table formatting into component
chore: add tsup build config
```

Prefixes: `feat`, `fix`, `refactor`, `chore`, `docs`, `test`

## Local Development

```bash
pnpm install          # install dependencies
pnpm dev              # run in development mode (ts-node or tsx)
pnpm build            # build with tsup
pnpm start            # run built version

# Test individual commands during dev:
pnpm dev -- scan
pnpm dev -- trade buy ETH-PERP 0.1 --testnet
pnpm dev -- init --testnet
```

## Build & Distribution

- `tsup` bundles TypeScript to `dist/cli.js`
- Package.json `bin` field maps `pacifica` to the built CLI
- `npm publish` for distribution
- Optional: `pkg` or `bun build --compile` for standalone binaries

## Testing

- Vitest if time allows
- Priority: test SDK client, guardrails logic, config parsing
- Manual testing against Pacifica testnet is the primary validation
- See context/developer/TESTING.md for strategy

## When to Commit

- After each command is working end-to-end
- After MCP server is serving tools
- Before starting risky refactors
- Before demo prep on Day 3
