## Data Flow

### Trading Flow (Human via CLI)
1. User runs `pacifica trade buy ETH-PERP 0.5 --lev 5 --tp 4000 --sl 3700`
2. Commander parses args → validates inputs
3. Core SDK sends REST request to Pacifica API
4. Response: order confirmation with fill price, fees
5. Journal logger appends trade to `.pacifica/journal.json`
6. Event system fires `on_fill` → hooks execute async
7. CLI prints formatted confirmation

### Trading Flow (Agent via MCP)
1. Agent calls `pacifica_place_order` MCP tool
2. Guardrails checker: Is agent enabled? Is action allowed? Within spending limit? Under confirmation threshold?
3. If guardrails pass → Core SDK sends REST request
4. Response returned as structured JSON to agent
5. Journal logger appends trade (tagged `triggered_by: agent`)
6. Event system fires `on_fill`
7. Agent log appends action to `.pacifica/agent-log.json`

### Live Data Flow
1. WebSocket connects to Pacifica on startup
2. Ticker updates stream in → update React state in Ink components
3. `pacifica scan` re-renders table on every tick
4. If disconnect > 5s → fallback to REST polling every 2s
5. On reconnect → resume WebSocket, stop polling

## Key Design Decisions

- See context/project/DECISIONS.md for full decision log
- Core library is shared between CLI, MCP server, and Skills — no duplication
- Guardrails enforcement happens in core, not at the CLI or MCP layer
- All write operations go through the same path regardless of trigger source
